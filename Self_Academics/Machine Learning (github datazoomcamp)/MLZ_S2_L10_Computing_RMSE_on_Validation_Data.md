# ML Zoomcamp — Session 2, Lesson 10: Computing RMSE on Validation Data

> [!info] Document Type
> **Type A — Applied ML / Data Engineering.** Synthesized from ML Zoomcamp Session 2 Lesson 10 (video transcript). Covers correcting the evaluation mistake from the baseline lesson: introducing the `prepare_x` helper to standardize feature preparation, then computing RMSE on the held-out validation set instead of the training set — establishing the correct two-phase evaluation architecture for all future experiments.

---

## 1. Executive Summary

Lesson 2.10 fixes a methodological flaw introduced in the baseline lesson: the RMSE was computed on the **same data used for training**, which gives an optimistically biased estimate of model quality. The correct procedure is to apply the trained model to the **validation set** — data the model has never seen — and measure RMSE there.

To make this clean, the lesson introduces a `prepare_x(df)` helper function that encapsulates the entire feature-matrix construction pipeline (column selection → zero-fill → `.values`). Because the exact same preparation steps must be applied consistently to training, validation, and test data, wrapping them in a single function eliminates duplication and prevents accidental inconsistencies.

The resulting architecture has two sharply separated phases:

1. **Training phase** — touches only `df_train`; produces weights `w0`, `w`.
2. **Validation phase** — applies the same `prepare_x` to `df_val`; predicts with trained weights; measures RMSE.

This two-phase structure is the foundation for every model improvement in Lessons 2.11–2.15.

## 2. Zero-Loss Extraction

### 2.1 The Evaluation Mistake (Lesson 2.8 Recap)

In Lesson 2.8, predictions were generated by applying the model to `X_train` — the same matrix used to train it:

```python
# WRONG — in-sample evaluation
y_pred = w0 + X_train.dot(w)
rmse(y_train, y_pred)
```

This computes **training error**, which is always optimistic. The model has already "seen" this data and has been fit to minimize its error on it. A model that merely memorizes training data will report low training RMSE while failing on new data.

The correct approach: compute RMSE on the **validation set**, which was held out during training (established in Lesson 2.4).

### 2.2 The `prepare_x` Function

The feature preparation pipeline consists of three steps that must be applied identically to every dataset (train, val, test). Extracting them into a function enforces this consistency:

```python
# Baseline numerical features (from Lesson 2.8)
base = ['engine_hp', 'engine_cylinders', 'highway_mpg', 'city_mpg', 'popularity']

def prepare_x(df):
    """
    Builds the NumPy feature matrix from a raw DataFrame.
    Applies identical preparation to any split (train / val / test).

    Steps:
      1. Select the five baseline numerical columns.
      2. Fill NaN values with 0.
      3. Extract and return the NumPy array.
    """
    df_num = df[base]
    df_num = df_num.fillna(0)
    X = df_num.values
    return X
```

This replaces what was previously three inline statements scattered across the training code.

### 2.3 Full Corrected Pipeline

```python
import numpy as np

# ── TRAINING PHASE ─────────────────────────────────────────────────────────────
# Only touches df_train. Produces the weights.

X_train = prepare_x(df_train)                            # (m_train, 5)
w0, w   = train_linear_regression(X_train, y_train)

# ── VALIDATION PHASE ───────────────────────────────────────────────────────────
# Applies the same preparation to df_val. Weights are NOT re-trained.

X_val   = prepare_x(df_val)                              # (m_val, 5)
y_pred  = w0 + X_val.dot(w)                              # predictions on val set

score   = rmse(y_val, y_pred)
print(f"Validation RMSE: {score:.4f}")
```

> **Key rule:** Nothing from `df_val` is used during the training phase. The weights `w0`, `w` are computed using only `df_train` and `y_train`.

### 2.4 The RMSE Function (from Lesson 2.9)

For completeness, the RMSE function used throughout:

```python
def rmse(y, y_pred):
    """Root Mean Squared Error."""
    error = y - y_pred          # residuals, shape (m,)
    se    = error ** 2          # squared residuals
    mse   = se.mean()           # mean squared error
    return np.sqrt(mse)         # RMSE
```

Mathematically:

$$\text{RMSE} = \sqrt{\frac{1}{m} \sum_{i=1}^{m} (\hat{y}_i - y_i)^2}$$

Because the target `y` is $\log(\text{price} + 1)$, RMSE is in log-price units. An RMSE of $0.5$ means predictions are off by roughly $e^{0.5} - 1 \approx 65\%$ in raw price.

### 2.5 Pipeline Architecture Overview

| Phase | Dataset Used | Operation | Output |
|---|---|---|---|
| Training | `df_train`, `y_train` | `prepare_x` + `train_linear_regression` | `w0`, `w` |
| Validation | `df_val`, `y_val` | `prepare_x` + matrix multiply + `rmse` | Validation RMSE score |
| (Future) Test | `df_test`, `y_test` | Same as validation | Final held-out RMSE |

## 3. Deep-Dive Explanations

### 3.1 Why In-Sample RMSE Is Optimistically Biased

The Normal Equation finds the **globally optimal weights** for the training data — weights that literally minimize $\sum_{i \in \text{train}} (\hat{y}_i - y_i)^2$. No other weight vector can achieve lower training error. This means:

- Training RMSE is a **lower bound** on validation/test RMSE.
- Any model with enough capacity can achieve near-zero training error by overfitting.
- Reporting training RMSE as the model's quality metric is misleading — it tells you how well the model memorized the data, not how well it generalizes.

The validation RMSE is the honest metric because the model had no influence over it during training.

### 3.2 The DRY Principle in ML Pipelines: Why `prepare_x` Matters

Without `prepare_x`, the feature-preparation code must be copy-pasted for every split:

```python
# Brittle — if you change one block and forget to change the others, bugs appear
X_train = df_train[base].fillna(0).values
X_val   = df_val[base].fillna(0).values    # easy to forget to update this
X_test  = df_test[base].fillna(0).values   # ...and this
```

With `prepare_x`, there is a single source of truth. When the feature set changes (e.g., Lesson 2.11 adds `age`), you modify `prepare_x` once and the change propagates to all splits automatically. This prevents **train-val-test skew** — one of the most common subtle bugs in ML codebases.

### 3.3 Validation Set vs. Test Set: What Gets Evaluated When

The three-split strategy from Lesson 2.4 assigns each split a distinct role:

| Split | Role | When accessed |
|---|---|---|
| Train | Fit model weights | Every training run |
| Validation | Tune model (features, hyperparameters) | During development — Lessons 2.10–2.14 |
| Test | Final honest evaluation | Once, after all decisions are made |

The validation set is used repeatedly throughout Sessions 2.10–2.14 to guide model improvement. This means by Lesson 2.14, the validation RMSE has been implicitly "used" in decisions — hence it too becomes a slightly optimistic estimate of the final model quality. The test set preserves a truly untouched benchmark.

### 3.4 Data Leakage: Why `prepare_x` Must Use Train-Side Statistics Only

If `prepare_x` computed a column mean for filling missing values (instead of zero), that mean **must** come from `df_train` only — never from `df_val` or `df_test`. Using validation or test statistics to inform preprocessing is called **data leakage**: the model implicitly "peeks" at future data, inflating validation RMSE optimism.

Because this lesson uses zero-filling (not mean-filling), there is no leakage — zero is a constant that requires no statistics from any split. Later lessons that introduce mean-filling or standardization must be careful about this.

## 4. Key Takeaways & Next Steps

| Concept | Rule |
|---|---|
| Training RMSE | Always optimistic — the Normal Equation minimizes it by construction |
| Validation RMSE | The honest metric — model never saw this data during training |
| `prepare_x(df)` | Single function for all splits — prevents train-val-test skew |
| Phase separation | Training phase touches only train data; validation phase uses trained weights on val data |
| Data leakage | Any statistics (mean, std) used in preprocessing must be computed on train set only |
| Zero-fill advantage | Requires no training-set statistics → no leakage risk |
